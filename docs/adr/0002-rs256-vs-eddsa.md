# ADR-0002: JWT 서명 알고리즘 — RS256 (vs EdDSA / ES256)

## 상태
적용

## 배경
JWT 서명 알고리즘 선택지는 크게 셋입니다.

- **HS256** (HMAC) — 대칭키. issuer 와 verifier 가 같은 비밀을 공유해야 하므로, 외부
  consumer 가 검증하는 IdP 시나리오에는 부적합.
- **RS256** (RSA-SHA256) — 비대칭. 표준이 가장 널리 깔려 있고, 모든 OAuth2 클라이언트가
  지원.
- **EdDSA / ES256** (Ed25519, ECDSA-P256) — 비대칭. 키 / 서명이 짧고 빠르지만, 일부
  오래된 클라이언트 호환성이 떨어짐.

## 결정
**RS256 / RSA 2048bit**.

이유:
1. 외부 consumer (다른 internal service, 모바일 앱 SDK, 향후 외부 파트너) 가 어떤 JWT
   라이브러리를 쓸지 모르므로 호환성 폭이 가장 넓은 RS256 을 기본값으로 둡니다.
2. RSA 2048 은 NIST 권고 (2030 까지 안전) 를 충족.
3. JWK header 의 `alg` 가 명시적이어서 verifier 가 알고리즘 혼동 공격 ({alg: none},
   {alg: HS256 with RSA pubkey}) 을 차단하기 쉬움.

## 다시 검토할 시점
- 모바일 SDK 에서 키 / 서명 크기가 문제가 될 때 (배터리 / 대역폭). EdDSA 를 추가하고
  `alg` 를 복수로 노출하면서 JWK rotation 으로 점진 전환.
- HSM / KMS 가 EdDSA 를 더 잘 지원하게 될 때.

## 결과
- JWKS endpoint 의 키 1개당 ~360 bytes (PEM 직렬화 기준).
- 서명 시간 ~1ms (M1, JIT 워밍업 후) — access token 발급 hot path 에 부담이 없습니다.
- (단점) 향후 EdDSA 로 전환할 때 클라이언트 라이브러리 호환성 매트릭스를 따로 관리해야
  합니다.

## 용어 풀이 (쉽게)

- **대칭키 vs 비대칭키** — 대칭키(HS256)는 발급자와 검증자가 '같은 비밀번호'를 나눠 가져야 해서, 검증하는 모두가 그 비밀을 알면 위조도 할 수 있다. 비대칭키(RS256)는 잠그는 열쇠(개인키)와 푸는 열쇠(공개키)가 달라, 검증자에겐 공개키만 줘도 돼 발급은 못 한다.
- **RS256 / RSA 2048bit** — RSA라는 비대칭 방식으로 SHA-256 서명을 만드는 알고리즘. 2048bit는 열쇠 길이로, 길수록 깨기 어렵다(NIST 권고상 2030년까지 안전).
- **EdDSA / ES256** — RSA보다 키·서명이 짧고 빠른 최신 비대칭 알고리즘. 좋지만 오래된 클라이언트는 아직 못 알아듣는 경우가 있다.
- **알고리즘 혼동 공격 (alg confusion)** — 검증기를 속여 서명 검사를 무력화하는 공격. '서명 없음(alg:none)'으로 위조하거나 공개키를 비밀번호처럼 착각하게 만든다. RS256으로 못박아 차단한다.
- **JWKS endpoint** — 토큰 검증에 쓸 '공개 열쇠 꾸러미'를 외부에 내려주는 주소. 검증자는 여기서 공개키를 받아 서명만 확인한다.
